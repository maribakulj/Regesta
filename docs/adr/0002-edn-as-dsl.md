# 0002 — EDN as the rule DSL, no parser

- Status: Accepted
- Date: 2026-04-17

## Context

The declarative rule layer is the center of gravity of MetaLisp. It must be
expressive enough to describe validation, inference, normalization, repair,
and projection logic across diverse metadata families, while remaining
amenable to inspection, serialization, composition, and eventual machine
assistance.

A bespoke text DSL (custom grammar, parser, error recovery) would give
designer freedom but adds a large implementation surface, a long road to
decent error messages, and a barrier to tooling. It also creates two
languages in the codebase (host + DSL) with their own mismatches.

## Decision

The rule DSL is **EDN**. A rule is a Clojure map with a fixed schema:

```clojure
{:id       :example/rule
 :phase    :validate
 :match    [[?r :record/kind :book]
            (absent? ?r :dc/title)]
 :produce  {:diagnostic {:severity :error
                         :code     :missing-title
                         :subject  ?r}}}
```

There is no parser. The rule compiler (`metalisp.rules`) reads EDN and
compiles rules into executable functions. The compiler validates rule shape
against a Malli schema; malformed rules fail at load time with a precise,
localized error.

Logic invoked from rules is restricted to a curated predicate stdlib. Rules
cannot call arbitrary Clojure functions: doing so would reopen the
"logic-buried-in-code" door we explicitly closed.

## Alternatives considered

- **Custom textual DSL with parser.** Rejected: large implementation cost,
  months of error-message polish, no natural advantage over EDN for our
  users (who read Clojure anyway).
- **Clojure macros as the rule language.** Rejected: macros are code, not
  data. They cannot be serialized, diffed, or authored by non-programmers.
- **Datalog source syntax (e.g. a string passed to a Datalog parser).**
  Rejected for the same reason as the bespoke DSL. We borrow Datalog's
  *shape* for pattern matching, expressed as EDN data.

## Consequences

- Rules are first-class data. They can be read from files, generated by
  other rules, inspected in the REPL, diffed in version control, and
  potentially produced or reviewed by tooling in the future.
- Zero-cost extensibility: new rule kinds require extending the compiler,
  not the parser.
- The curated predicate stdlib becomes an important design artifact in its
  own right. Growing it carelessly is the main way this decision could turn
  sour — each addition should be weighed as if it were a new keyword.
- Users must tolerate EDN syntax. For the target audience (engineers and
  metadata specialists willing to learn a structured format) this is fine;
  purely non-technical authors will need a higher-level editor, which is an
  explicit non-goal of V1.
